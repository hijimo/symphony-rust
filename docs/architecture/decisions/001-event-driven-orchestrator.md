# ADR-001: 事件驱动单线程编排器

## 状态

accepted

## 上下文

Symphony 的编排器需要同时管理多个并发运行的 Agent Worker，每个 Worker 对应一个 Issue 的执行任务。编排器需要处理以下复杂场景：

- Worker 异常退出后的重试调度
- 定时轮询 Tracker 发现新 Issue
- 实时接收 Codex 事件流（token 统计、活动时间更新）
- 响应 HTTP API 的状态查询请求
- 处理配置热重载通知
- 执行停滞检测和终态协调

这些操作涉及共享状态（`running` map、`claimed` set、`retry_attempts` 等），需要在并发环境下保证状态一致性。

核心约束：
- 状态变更必须是原子的，不能出现部分更新
- 调试和推理必须简单，避免难以复现的并发 bug
- 性能要求不高（编排器本身不是瓶颈，瓶颈在 Agent 执行）

## 决策

采用**单线程事件循环**模式：

- 所有状态变更通过单一 `tokio::sync::mpsc` channel 序列化
- 编排器主循环是唯一持有和修改状态的执行上下文
- Worker 任务通过 channel 异步上报事件，不直接访问共享状态
- HTTP API 通过 `oneshot` channel 向编排器发送查询请求，等待回复

```
Worker 1 ──┐
Worker 2 ──┤  mpsc channel  ┌─────────────────────┐
Worker 3 ──┤ ─────────────► │  Orchestrator        │
HTTP API ──┤                │  单线程事件循环        │
定时器    ──┘                │  所有状态变更序列化    │
                            └─────────────────────┘
```

## 备选方案

### 方案 A：Actix Actor 模型

使用 Actix 框架，将编排器建模为 Actor，每个 Worker 也是独立 Actor，通过消息传递通信。

未采用原因：
- Actix 引入较重的框架依赖，与项目已有的 Tokio 生态不完全契合
- Actor 模型的消息路由和监督树增加了概念复杂度
- 对于本项目的规模，Actor 模型的优势（分布式、容错树）并不需要

### 方案 B：RwLock 共享状态

使用 `Arc<RwLock<OrchestratorState>>` 在多个任务间共享状态，读操作并发，写操作独占。

未采用原因：
- 多处持有锁增加死锁风险，尤其是跨 await 点持锁
- 锁竞争在高并发写场景下成为性能瓶颈
- 状态变更的原子性难以保证（多个字段需要同时更新）
- 调试并发问题（死锁、数据竞态）成本高

### 方案 C：数据库作为状态存储

将编排状态持久化到 SQLite，通过数据库事务保证一致性。

未采用原因：
- 编排状态是高频读写的内存数据，数据库 I/O 会成为瓶颈
- 增加了外部依赖和故障点
- 编排状态本质上是运行时临时状态，重启后通过重新轮询 Tracker 恢复即可

## 后果

正面影响：
- 状态一致性保证简单：任意时刻只有一个执行上下文修改状态
- 无死锁风险：不存在锁，不存在循环等待
- 调试容易：事件按序处理，状态变更可追溯
- 代码简洁：状态访问无需加锁，逻辑清晰

负面影响：
- 单点序列化：所有事件串行处理，高频事件（如大量 CodexUpdate）可能造成队列积压
- 不适合 CPU 密集型操作：编排器主循环不能执行耗时计算，否则会阻塞事件处理
- 状态不持久化：重启后丢失重试计数和运行时统计（已接受的权衡）

缓解措施：
- `CodexUpdate` 事件处理逻辑保持轻量（仅更新计数器和时间戳）
- 耗时操作（Tracker API 调用、文件 I/O）在独立 Tokio 任务中执行，结果通过 channel 回传
