# ADR-003: 进程管理器 Lease/Generation 设计

## 状态

accepted

## 上下文

web-platform 需要可靠地管理 rust-platform 子进程的完整生命周期：启动、停止、健康监控、异常恢复。这个问题比表面看起来复杂，存在以下挑战：

**多实例竞争**：如果 web-platform 意外重启，新实例可能发现数据库中记录了一个"运行中"的进程，但不确定该进程是否真的属于自己管理。

**孤儿进程**：web-platform 崩溃后，rust-platform 子进程可能继续运行，成为孤儿进程。下次启动时需要能够识别并清理这些孤儿进程。

**PID 复用**：操作系统会复用 PID。仅凭 PID 无法确认进程身份——数据库中记录的 PID 可能已经被另一个完全不相关的进程占用。

**并发操作**：用户可能快速连续点击"停止"和"启动"，需要防止操作交叉导致状态混乱。

**服务代次**：需要区分"第一次启动的进程"和"重启后的新进程"，避免旧进程的状态污染新进程的管理。

## 决策

采用**三层 Lease 标识**机制：

```
web_instance_id
    — web-platform 实例的唯一 ID
    — 服务启动时生成（UUID v4），持久化到数据库
    — 用于识别"这个进程是由哪个 web-platform 实例启动的"

lifecycle_op_id
    — 每次启动/停止操作的唯一 ID
    — 操作发起时生成（UUID v4）
    — 用于防止并发操作交叉（乐观锁语义）

service_generation
    — 服务代次，每次成功启动后递增
    — 用于区分同一 web-platform 实例的多次启动
    — 结合 web_instance_id 唯一标识一次服务运行
```

工作流程：

```
启动操作：
1. 生成新的 lifecycle_op_id 和 service_generation
2. 将三层 ID 写入数据库（pending 状态）
3. 将三层 ID 作为环境变量传递给子进程
4. 子进程启动后，将自身 PID + 三层 ID 写回数据库（running 状态）

心跳监控：
1. 定期读取数据库中的 PID 和三层 ID
2. 用 kill -0 验证 PID 是否存活
3. 验证进程环境变量中的三层 ID 是否与数据库一致
4. 不一致 → 触发孤儿清理

孤儿清理：
1. 扫描数据库中所有"running"状态的进程记录
2. 对每条记录执行 PID + Lease 验证
3. Lease 不匹配的进程视为孤儿，发送 SIGTERM
4. 超时后发送 SIGKILL
```

## 备选方案

### 方案 A：简单 PID 跟踪

仅在数据库中记录子进程 PID，通过 `kill -0` 检查进程是否存活。

未采用原因：
- PID 复用问题：操作系统回收 PID 后可能分配给其他进程，`kill -0` 返回成功但进程已不是 rust-platform
- 无法区分孤儿进程和正常进程
- 多实例场景下无法确认进程归属

### 方案 B：systemd 委托

将 rust-platform 的生命周期管理委托给 systemd，web-platform 通过 D-Bus 或 systemctl 控制服务。

未采用原因：
- 强依赖 systemd，不支持 macOS 和非 systemd Linux 发行版
- 增加系统级配置复杂度（需要 systemd unit 文件、权限配置）
- 与"单二进制零配置部署"目标相悖
- 开发环境调试不便

### 方案 C：文件锁（flock）

子进程持有一个锁文件，web-platform 通过尝试获取锁来判断进程是否存活。

未采用原因：
- 文件锁在进程崩溃时会自动释放，但无法携带身份信息
- 无法解决多实例竞争问题
- 无法区分服务代次

## 后果

正面影响：
- 多实例安全：即使 web-platform 意外重启，新实例也能正确识别孤儿进程
- 孤儿检测可靠：三层 ID 组合唯一标识一次服务运行，PID 复用不会造成误判
- 并发操作安全：`lifecycle_op_id` 提供乐观锁语义，防止操作交叉
- 可审计：数据库中保留完整的启动历史和代次信息

负面影响：
- 实现复杂度增加：相比简单 PID 跟踪，需要维护三层 ID 的生成、传递、验证逻辑
- 子进程需要配合：rust-platform 子进程需要在启动时将 Lease 信息写回数据库
- 调试难度略增：排查进程管理问题时需要理解三层 ID 的语义

已接受的权衡：
- 复杂度增加是值得的，换来了可靠的多实例安全和孤儿检测能力
