# ADR-002: 使用 SQLite 作为 web-platform 存储

## 状态

accepted

## 上下文

web-platform 需要持久化存储以下数据：
- 用户账户（用户名、密码哈希、角色）
- 项目配置（平台类型、仓库地址、加密 Token）
- Issue 和看板数据
- 项目成员关系
- 告警规则和历史
- 进程管理 Lease 信息（PID、web_instance_id、service_generation）

系统的部署目标是**单机部署**，不需要多节点水平扩展。典型用户规模：数十个项目、数百个 Issue、少量并发用户（团队内部工具）。

核心约束：
- 部署必须简单，零外部依赖
- 备份和迁移必须简单
- 不需要支持多节点部署
- 写并发量低（管理操作，非高频写入）

## 决策

使用 **SQLite + r2d2 连接池 + Refinery 迁移**：

- `rusqlite`（bundled feature）：将 SQLite 静态链接到二进制，无需系统安装 SQLite
- `r2d2` + `r2d2_sqlite`：连接池管理，支持并发读写
- `refinery`：版本化数据库迁移，迁移文件随代码版本管理，服务启动时自动执行

```
web-platform 二进制
    └── SQLite（静态链接，bundled feature）
            └── data.db（单文件，可直接 cp 备份）
```

## 备选方案

### 方案 A：PostgreSQL

使用 PostgreSQL 作为关系型数据库，通过 `sqlx` 或 `diesel` 访问。

未采用原因：
- 需要独立安装和运行 PostgreSQL 服务，增加部署复杂度
- 需要配置连接字符串、用户权限、数据库初始化
- 备份需要 `pg_dump`，迁移需要额外工具
- 对于团队内部工具的规模，PostgreSQL 的能力远超需求
- 与"零外部依赖单机部署"目标相悖

### 方案 B：嵌入式键值存储（sled / RocksDB）

使用嵌入式键值存储替代关系型数据库。

未采用原因：
- 数据模型天然是关系型的（项目-成员、Issue-MR 等），键值存储需要手动维护关系
- 缺乏 SQL 查询能力，复杂查询（过滤、排序、聚合）需要手动实现
- 生态和工具链不如 SQLite 成熟
- 迁移和 Schema 演进没有标准方案

### 方案 C：纯内存存储

将所有状态保存在内存中，服务重启后丢失。

未采用原因：
- 用户账户、项目配置等数据必须持久化
- 服务重启（如 web-platform 升级）不能丢失数据
- 无法支持多次部署间的数据延续

## 后果

正面影响：
- 零外部依赖：SQLite 静态链接，单二进制部署，`cp data.db` 即可完成备份
- 部署简单：无需配置数据库服务，开箱即用
- 迁移简单：Refinery 自动执行版本化迁移，Schema 演进有记录
- 事务支持：SQLite 支持完整的 ACID 事务，数据一致性有保障
- 工具链成熟：可用标准 SQLite 工具（sqlite3 CLI、DB Browser）直接查看和操作数据

负面影响：
- 写并发限制：SQLite 写操作串行化，高并发写入场景性能受限（对本项目不是问题）
- 不支持水平扩展：无法多节点共享同一数据库（对本项目不是需求）
- WAL 模式注意事项：需要正确配置 WAL 模式以支持并发读写，否则读写会互相阻塞

已接受的权衡：
- 写并发限制在团队内部工具场景下完全可接受
- 单机部署限制符合当前产品定位
