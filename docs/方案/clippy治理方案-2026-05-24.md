# Clippy 治理方案

## 背景

Issue #13 用于记录 main 分支基线验证失败后的 Clippy 治理方案。当前要求是不在现有业务 PR 中处理这些 Clippy 问题，先把方案沉淀到本地文档，再经对抗验证后决定后续拆分。

当前 main 已恢复基础验证基线：

- `cargo fmt -- --check`
- `cargo test`

但 `cargo clippy --all-targets --all-features -- -D warnings` 仍有既有失败。Strict Clippy 暂不应直接恢复为默认阻塞 gate，否则会把既有技术债转嫁给无关业务 PR；但默认 CI 也不应完全没有 Clippy gate，否则治理期间会继续引入新债务。

## 当前 Clippy 失败分类

### too_many_arguments

- `rust-platform/src/agent/runner.rs::run_turn_loop`
- `rust-platform/src/orchestrator/retry.rs::schedule_retry`
- `web-platform/src/handlers/project_service.rs::build_lifecycle_update`
- `web-platform/src/repository/traits.rs` 中 repository trait 方法

风险判断：

- 不能把这一类作为一个机械“参数对象 PR”处理。
- 局部函数和 repository trait 是不同风险边界。
- 局部函数也不应强制引入 context/request struct；可以拆函数、合并已有上下文、移除派生参数，或在确有领域聚合时引入小 struct。
- repository trait 是抽象边界，改动会影响 trait、SQLite 实现、handler 调用、mock 和测试，应单独设计持久化层语义的 query/input struct。

### large_enum_variant

- `rust-platform/src/models/mod.rs::OrchestratorEvent::CodexUpdate`

风险判断：

- 可以提前单独处理。
- 建议只 boxing `OrchestratorEvent::CodexUpdate` 的 `update` 字段。
- 不要顺手改 producer API、事件通道类型、事件模型命名或 Codex update 结构。

### await_holding_lock

- `rust-platform/src/agent/codex_client.rs` 测试中的 env lock
- `rust-platform/src/proxy.rs` 测试中的 env lock
- `rust-platform/src/workspace/mod.rs` 测试中的 env lock

风险判断：

- 不能简单缩短 std `MutexGuard` 生命周期。
- 这些锁保护的是进程级环境变量，跨 await 保持互斥本身有测试隔离意义。
- 简单 drop guard 可能让并发测试重新污染 env。
- 更稳方向是统一 test-only env guard/helper，由 helper 持锁、修改 env、恢复 env，并禁止相关测试绕过 helper 直接 `set_var/remove_var`。

### should_implement_trait

- `web-platform/src/models/alert.rs::Severity::from_str`
- `web-platform/src/models/project.rs::ServiceStatus::from_str`
- `web-platform/src/notification/mod.rs::Severity::from_str`

风险判断：

- 不应机械统一实现 `FromStr`。
- 三处语义不同：有默认降级、有 `Option`，且类型分属不同领域。
- 更保守做法是逐处确认语义后改成明确方法名，例如 `parse_or_default`、`from_label`，只有语义明确时再实现标准 `FromStr`。
- 这一项不应和纯机械的 test module 移动混在一个 PR。

### items_after_test_module

- `web-platform/src/handlers/network_proxy.rs`

风险判断：

- 低风险。
- 可移动 test module 到文件末尾，或拆出测试模块。

## 治理原则

1. 不在业务 PR 中清理 #13 的 Clippy 技术债。
2. 不用 `#[allow(clippy::...)]` 作为长期方案。
3. 治理期间必须防止新增 Clippy diagnostic。
4. 每个治理 PR 只处理一个清晰风险边界。
5. 恢复 strict Clippy CI 的 PR 只改 CI 和必要文档，不夹带代码修复。
6. live GitLab `#[ignore]` 与 Clippy 治理解耦；它属于外部环境测试策略问题。

## 过渡 CI Gate 策略

### 主线方案：Baseline-aware Clippy Gate

PR 0 必须以 baseline-aware gate 作为主线方案。目标是阻止 baseline 之外的新增 Clippy diagnostic，同时不强迫无关业务 PR 清理既有债务。

命令基础：

```bash
cargo clippy --all-targets --all-features --message-format=json -- -D warnings
```

实现要求：

- CI wrapper 必须捕获 `cargo clippy` 的原始退出码，继续解析 JSON 输出；既有 baseline diagnostic 会导致 Clippy 返回非零，但最终 CI 结果只能由 baseline diff 决定。
- baseline-only diagnostic 时，wrapper 最终返回 0；baseline 外新增 diagnostic、baseline 文件无效、baseline 未排序、或 stale entry 未处理时，wrapper 最终返回非 0。
- 解析 JSON 输出时只处理 diagnostic，过滤 compiler-artifact 等非诊断消息。
- baseline 文件必须结构化、排序稳定，并纳入版本管理。
- baseline 文件需要有明确生成和校验命令，例如：

```bash
cargo clippy --all-targets --all-features --message-format=json -- -D warnings \
  | scripts/clippy-baseline normalize --stdin > target/clippy-diagnostics.json
scripts/clippy-baseline check --baseline .clippy-baseline.json --diagnostics target/clippy-diagnostics.json
```

- baseline entry 至少包含：
  - lint code
  - crate/package
  - target kind 或 all-targets 维度
  - feature set
  - primary span 文件路径
  - 从 primary span 或 diagnostic spans 稳定提取的 item/function/type；如果无法稳定提取，必须为空并由 fingerprint 区分，不能使用人工摘要
  - diagnostic message 摘要
  - fingerprint algorithm version
  - 同文件同 message 多处出现时的 occurrence index 或稳定 fingerprint
- baseline key 尽量减少对精确行号的依赖，但必须能区分同文件同 lint 的多处诊断。
- macro 展开、多 span diagnostic 以 primary span 为主；无 primary span 时必须降级为明确的 fallback key，并在 summary 中标注。
- 文件移动或重命名时不得自动扩大 baseline；治理 PR 如果移动代码，应同步更新或减少 baseline，并在 PR 描述中说明。
- baseline 只能减少或等量迁移，不允许无解释增加。
- baseline check 必须检测 stale entry：baseline 中存在但当前 diagnostics 不再出现的 entry 必须要求删除，除非该 PR 明确说明为等量迁移。
- baseline 文件排序、fingerprint algorithm version、target/features 维度必须由脚本校验，避免人工编辑造成不可重复匹配。

CI 输出要求：

- job log 或 artifact 必须稳定输出：
  - 原始 Clippy JSON
  - 规范化 diagnostics JSON
  - baseline diff JSON
  - 新增 diagnostic 列表
  - 命中的 baseline 列表或数量
  - baseline 剩余数量 diff
  - 本次减少的 diagnostic
- 不能只把 strict Clippy 剩余清单放在 PR 描述里。

PR 0 负向验证：

- 临时构造一个 baseline 外的同类 Clippy warning，验证 job 必须失败。
- 临时构造一个不同 lint code 的新增 warning，验证 job 必须失败。
- 临时构造一个同 lint code、同文件、同 message 的新增 occurrence，验证 job 必须失败。
- 使用未修改代码运行 baseline-only diagnostics，验证 job 必须通过，即使 `cargo clippy` 原始退出码非 0。
- 验证失败输出应能定位新增 lint code、文件、item/message 摘要。
- 负向验证代码不进入最终提交。

### 临时附录：Allow-list Stopgap

如果交付压力要求先恢复最基本 Clippy 覆盖，可短期使用 allow-list gate：

```bash
cargo clippy --all-targets --all-features -- \
  -D warnings \
  -A clippy::too_many_arguments \
  -A clippy::large_enum_variant \
  -A clippy::await_holding_lock \
  -A clippy::should_implement_trait \
  -A clippy::items_after_test_module
```

限制：

- 只能称为“防新增 lint 类别”，不能称为“防新增 Clippy debt”。
- 不能阻止既有五类 lint 下新增 diagnostic。
- 不能作为正式 PR 0 的替代品。
- 必须同时登记 baseline-aware gate 的后续任务、负责人和短截止日期。
- 业务 PR 若新增 allow-list 内问题，CI 可能不会拦截，review 规则仍应阻止。
- 不得新增 `#[allow(clippy::...)]` 来扩大豁免面。

## 推荐 PR 拆分和顺序

### PR 0：Baseline-aware Clippy Gate

目标：

- 默认 CI 增加 baseline-aware Clippy gate。
- 不改业务代码。
- 阻止 baseline 外新增 Clippy diagnostic。

验收：

- `cargo fmt -- --check`
- `cargo test`
- baseline-aware Clippy gate 通过
- baseline-only diagnostics 时 gate 通过，即使 `cargo clippy` 原始退出码非 0
- 负向验证证明 baseline 外新增同类 diagnostic 会失败
- 负向验证证明新增不同 lint code diagnostic 会失败
- 负向验证证明同 lint code、同文件、同 message 的新增 occurrence 会失败
- baseline 生成、排序校验、stale entry 检测均通过
- CI job log 或 artifact 输出新增项、baseline 命中项、剩余数量 diff、减少项

### PR 1：Large Enum Variant

范围：

- `OrchestratorEvent::CodexUpdate`

要求：

- 只 boxing `update: Box<CodexEventUpdate>` 及直接构造/消费点。
- 不扩大 producer API。
- 不重塑事件模型。
- 不混入其它 cleanup。

验收：

- `cargo fmt -- --check`
- `cargo test -p symphony-platform`
- baseline-aware Clippy gate 通过
- baseline diff 显示 `large_enum_variant` 对应 entry 减少，且无新增 diagnostic

### PR 2：Items After Test Module

范围：

- `web-platform/src/handlers/network_proxy.rs`

要求：

- 只移动 test module 到文件末尾，或拆出测试模块。
- 不改运行时代码语义。
- 不混入 `from_str` 或其它 lint cleanup。

验收：

- `cargo fmt -- --check`
- `cargo test -p web-platform`
- baseline-aware Clippy gate 通过
- baseline diff 显示 `items_after_test_module` 对应 entry 减少，且无新增 diagnostic

### PR 3：Should Implement Trait

范围：

- `web-platform/src/models/alert.rs::Severity::from_str`
- `web-platform/src/models/project.rs::ServiceStatus::from_str`
- `web-platform/src/notification/mod.rs::Severity::from_str`

要求：

- 三处逐项确认现有语义。
- 不机械统一 `FromStr`。
- 必须覆盖默认降级、`Option`、标签解析等现有行为。
- 方法改名或 trait 实现后，调用点语义保持不变。

验收：

- `cargo fmt -- --check`
- `cargo test -p web-platform`
- 覆盖 `alert::Severity` 默认降级语义
- 覆盖 `ServiceStatus` 默认降级语义
- 覆盖 `notification::Severity` 的 `Option` 语义
- baseline-aware Clippy gate 通过
- baseline diff 显示 `should_implement_trait` 对应 entries 减少，且无新增 diagnostic

### PR 4：局部 Too Many Arguments

范围：

- `run_turn_loop`
- `schedule_retry`
- `build_lifecycle_update`

要求：

- 优先保持局部、行为不变。
- 可拆函数、合并已有上下文、移除派生参数，或在确有领域聚合时引入小 struct。
- 如果引入 struct，struct 只服务当前模块或函数族，不跨 agent、orchestrator、web handler 等领域共享。
- 不引入公共 “Context” 大包。
- 不碰 repository trait。

验收：

- `cargo fmt -- --check`
- `cargo test -p symphony-platform`
- `cargo test -p web-platform`
- 相关 orchestrator/agent/web handler 测试通过
- baseline-aware Clippy gate 通过
- baseline diff 显示局部函数 `too_many_arguments` entries 减少，且无新增 diagnostic

### PR 5：Test-only Env Guard

范围：

- `rust-platform/src/agent/codex_client.rs`
- `rust-platform/src/proxy.rs`
- `rust-platform/src/workspace/mod.rs`

要求：

- 统一 test-only env guard/helper。
- helper 负责持锁、保存 env、设置 env、恢复 env。
- helper 不得持有 `std::sync::MutexGuard` 跨 await。
- 如需跨 await 保持互斥，必须使用 async-aware guard、串行测试封装，或闭包式 helper 让 async 区间处于明确的 test-only 串行边界内。
- 相关测试不得直接 `std::env::set_var/remove_var`。
- 锁作用域覆盖所有相关 env 读写和跨 await 区间。
- 不用简单 drop guard 换取 Clippy 通过。

验收：

- 相关 package tests 通过。
- 目标 ignored 测试可单独运行。
- 并发运行相关测试时 env 不串扰。
- strict Clippy 中 `await_holding_lock` 对应项消失。
- baseline-aware Clippy gate 通过，且无新增 diagnostic。

### PR 6：Repository Trait 参数对象

范围：

- `ProjectRepository::list_projects_for_user`
- `ConcurrencyRepository::record_concurrency_event`

要求：

- request/event struct 必须是持久化层语义的 query/input。
- 禁止复用 handler DTO、UI 请求模型或上层 event。
- 同步 trait、SQLite impl、调用点、mock 和相关测试。
- 不得混入 schema migration。
- 不得改变 SQL 查询语义、分页、排序、过滤、默认值或权限边界。
- 不和业务功能混在一起。

验收：

- trait mock/impl/caller 编译通过。
- SQLite impl 相关测试通过。
- handler 调用路径相关测试通过。
- baseline 对应 `too_many_arguments` 项减少。

### PR 7：恢复 Strict Clippy CI

范围：

- CI 配置
- 必要说明文档

要求：

- 默认 CI 恢复 `cargo clippy --all-targets --all-features -- -D warnings`。
- 移除临时 allow-list 或 baseline 机制。
- 不夹带代码修复。

进入条件：

- main 上本地 strict Clippy 已经 0 warning。
- CI strict Clippy 已经 0 warning。
- baseline 已无剩余条目，或所有剩余条目已由前序 PR 清理。

## 业务 PR 规则

默认规则：

- 业务 PR 不处理 #13 Clippy debt。

唯一例外：

- 与业务改动同文件、同局部函数。
- 行为不变。
- 不涉及 CI、公共 trait、事件 enum、测试基础设施、repository 边界。
- 不扩大 diff。
- PR 描述必须显式声明该例外，并列出验证命令。

除此之外，一律拆独立治理 PR。

## 禁止混入业务 PR 的内容

- repository trait 签名调整
- request/context struct 大范围引入
- `OrchestratorEvent` enum 形状变化
- env lock 或测试基础设施变化
- Clippy gate / baseline 机制变化
- 恢复 strict Clippy CI

## Live GitLab 测试说明

live GitLab `#[ignore]` 与 Clippy 治理解耦：

- `#[ignore]` 只影响默认 `cargo test` 是否运行外部环境测试。
- `cargo clippy --all-targets` 仍会检查 ignored 测试代码。
- live GitLab 覆盖应通过独立的 manual 或 scheduled secret-backed smoke job 处理。
- live GitLab smoke job 不应作为恢复 strict Clippy 的前置条件。

live smoke job 最小契约：

- 由 `workflow_dispatch` 或 scheduled job 触发。
- 需要真实 secret 时，secret 缺失应明确 skip 或 fail-fast，不能伪装通过。
- 是否阻塞 main 应与默认 `cargo test` 明确区分。
- 失败归类为外部集成回归，不混入 Clippy gate。

## 当前结论

原始 #13 方案的大方向可接受：先恢复 `fmt + test`，再拆小 PR 治理 Clippy，最后恢复 strict gate。

但方案必须按两轮对抗验证结论修订后再推进：

1. PR 0 主线必须是 baseline-aware Clippy gate，allow-list 只能作为短期 stopgap。
2. PR 顺序调整为：baseline gate -> large enum -> test module -> from_str -> 局部参数治理 -> env guard -> repository trait -> strict gate。
3. `too_many_arguments` 必须拆成局部函数治理和 repository trait 治理。
4. env lock 必须通过统一 test-only env guard/helper 维护测试隔离语义。
5. 业务 PR 默认不处理 #13，例外必须窄口径并显式声明。
